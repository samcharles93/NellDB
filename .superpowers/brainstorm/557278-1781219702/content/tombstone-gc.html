Tombstone Garbage Collection

Pruning the append-only log to prevent bloat

In a distributed system, you can't just delete a record. You write a Tombstone (Deleted: true) so other nodes know it's gone. But these tombstones can't stay forever.

A

Strict TTL (Default 30 Days)

Simple and predictable. After 30 days, the tombstone is physically deleted. Risk: A node offline for 31 days might "revive" the data when it syncs.

B

Knowledge Vector Quorum

Safest. Only delete a tombstone when the KnowledgeVector proves that every known node has seen an HLC greater than the tombstone's clock.

C

Manual / Snapshotting

Users trigger a "compact" operation. The system creates a snapshot and wipes the log. Best for high-write workloads where you want control over IO.

The "Zombie" Problem (TTL Risk)
Node A
Deletes Doc X
(Tombstone)
➔ 30 Days Pass ➔
Node B
Wakes up after 40 days
"Hey, I have Doc X!"

Node A has deleted the tombstone, so it thinks Node B's version is a new write.
