Security Rules Architecture

Adopting a Firestore-style declarative model

To support Document -> Collection -> Global rules without reinventing the wheel, we can use a declarative approach. Here are three ways to implement it in Go.

A

CEL (Common Expression Language)

The industry standard (used by Firestore/Envoy). Use google/cel-go to evaluate rules. Very powerful, supports complex logic on request and resource.

allow write: if request.auth.uid == resource.data.ownerId;
B

Simplified JSON/YAML Rules

Easier to parse and ship in a tiny binary. Map-based lookups with basic operators (==, in, exists).

rules:
  collections:
    messages:
      write: "auth.uid == data.sender"
C

Go Plugin / Middleware Hooks

Define rules as Go functions. Maximum performance, but requires re-compiling or using Go plugins. Hardest to sync across a distributed mesh.

Why Firestore-style (CEL) Wins

  • Familiarity: Devs already know the syntax.
  • Security: CEL is non-Turing complete (won't hang your server).
  • Portability: Rules can be synced between Go server and WASM client.
